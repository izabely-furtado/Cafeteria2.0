Diagrama do Padrão Decorador

O projeto foi feito com base em Cafeteria, no link está representado os dados referentes aps padrões anteriormente utilizados, como o padrão criativo fábrica.

Queremos que, dado um objeto Cafe, seja possível adicionar funcionalidades a ele, e somente a ele, em tempo de execução. Vamos ver a arquitetura sugerida pelo padrão decorador com o diagrama que o representa:

Explicação

Certo, então todos os objetos possuem o mesmo tipo Cafe, esta classe define o que todos os objeto possuem e é igual a classe já feita antes. As classes de cafés concretas definem apenas os dados relativos a ela.

Vejamos o construtor da classe abstrataFacricaCafe :

  FabricaCafe(float preco){
    //setando o preço
    this.setPreco(preco);
    //colocando os ingredientes
    this.agua = this.criarAgua();
}

Para ter um melhor parâmetro do que cada filho vai setar, segue como exemplo o construtor da classe FabricaCafeExpresso : 

private FabricaCafeExpresso() {
    super(2);
    this.acucar = this.criarAcucar();
    this.cafeSoluvel = this.criarCafeSoluvel();
    this.leiteLiq = this.criarLeiteLiq();
}

Já na classe abstrata do decorador temos o seguinte construtor:

public CafeDecorator(FabricaCafe cafe, float preco) { super(preco); this.umCafe = cafe; this.setPreco(this.getPreco() + this.umCafe.getPreco());

}

Assim em cada classe concreta de complemento tem-se que está vai ter passado como parâmetro um FabricaCafe e este estará contido no complemento. Já o preço, o preço do complemento em si é somado ao preço do café já escolhido.

Motivação

  • Adiciona funcionalidades ao objeto em tempo de execução. Note bem que, ao contrário da herança que aplica funcionalidades a todos os objetos dela, o padrão decorator permite aplicar funcionalidades apenas a um objeto específico.

  • Possui uma flexibilidade maior que a herança estática.

  • Aplica apenas as funcionalidades necessárias ao objeto nós evitamos o problema de classes sobrecarregadas, que possuem funcionalidade que nunca são utilizadas.
